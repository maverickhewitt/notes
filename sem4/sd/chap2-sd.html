4.1 SDLC Methodology

A methodology is a formalized approach to implementing the Software Development Life Cycle (SDLC). Various methodologies include:

  • Waterfall Development: A linear sequential model where each phase must be completed before the next begins.
  • Parallel Development: Multiple phases are developed simultaneously to save time.
  • V-Model: An extension of the waterfall model that emphasizes verification and validation at each stage.
  • Rapid Application Development (RAD): Focuses on rapid prototyping and user feedback.
  • Iterative Development: Development is done in small, iterative cycles, allowing for adjustments based on feedback.
  • System Prototyping: Creating prototypes to visualize and test system functionality early in the development process.
  • Throwaway Prototyping: Building a prototype to understand requirements, which is discarded after use.
  • Agile Development: An iterative approach that promotes flexible responses to change and customer collaboration.
  • Object-Oriented Development: Focuses on using objects to represent data and methods.

4.1.4 Rapid Application Development (RAD)

RAD is a collection of methodologies aimed at fast system development. Key features include:

  • Incorporation of special techniques and computer tools to speed up phases.
  • Delivery of portions of the system to users for evaluation and feedback.
  • Use of CASE (Computer-Aided Software Engineering) tools, JAD (Joint Application Development) sessions, and fourth-generation programming languages (e.g., Visual Basic.NET).
  • Increased user expectations can lead to scope creep or feature creep.

Agile Development

Agile methodologies emphasize flexibility and customer collaboration. Key principles include:

  • Continuous testing.
  • Simple coding performed by pairs of developers.
  • Close interactions with end users to build systems quickly.

SCRUM Methodology

SCRUM is a high-intensity teamwork methodology characterized by:

  1. Product Backlog: A prioritized list derived from product specifications.
  2. Sprint Backlog: A dynamic list of tasks to be completed in the next sprint.
  3. Sprint: A 30-day period where the development team transforms the backlog into demonstrable software.
  4. Daily Scrum: A brief meeting for team members to discuss progress, obstacles, and plans.
  5. Demo: Working software that can be demonstrated to the customer.

4.1.6 Object-Oriented Systems Analysis and Design

Object-oriented analysis combines data and processes into objects, facilitating rapid system changes. Key points include:

  • Systems can be added gradually, one subsystem at a time.
  • Reuse of previously written software is possible.
  • Unified Modeling Language (UML) is the industry standard for modeling object-oriented systems.

4.1.7 Selecting the Appropriate Development Methodology

Factors to consider when selecting a development methodology include:

  • Structured Methodologies:
    • Documentation of each step is crucial.
    • Management prefers the safety of a defined SDLC.
    • Adequate resources and time are available for full SDLC completion.
    • System requirements can be clearly defined early on.
  • Agile and RAD Methodologies:
    • Quick development is necessary in a dynamic environment.
    • Customer satisfaction with incremental improvements is essential.
    • Requires experts in agile/RAD to lead projects.
    • Agreement on agile principles among executives and analysts is needed.
  • Object-Oriented Methodologies:
    • Problems can be modeled using classes and objects.
    • Support for UML learning within the organization.
    • Gradual addition of system parts is feasible.
    • Reuse of existing software is encouraged.
    • Ability to tackle difficult problems first is acceptable.

4.2 System Analysis Phase Tasks

Key tasks in the system analysis phase include:

  • Requirements Modeling:
    • Fact-finding to describe the current system.
    • Identification of new system requirements.
    • Outputs: Information produced by the system.
    • Inputs: Necessary data entering the system.
    • Processes: Logical rules transforming data into information.
    • Performance: Speed, volume, capacity, availability, reliability.
    • Security: Controls safeguarding the system and its data.
  • Data and Process Modeling:

    Graphical representation of system data and processes using structured analysis techniques.

  • Object Modeling:

    Combining data and processes into objects for analysis.

  • Development Strategies:

    Preparation for transitioning to the system design phase.

Modeling Tools and Techniques

Modeling involves graphical methods and non-technical language to represent the system. Key tools include:

  • Data Flow Diagram (DFD)
  • Unified Modeling Language (UML)
  • Use Case Diagrams
  • Sequence Diagrams

4.3 System Requirements Checklist

System requirements are characteristics or features that must be included in an information system. Categories include:

  • Inputs: Examples include data entry requirements and uniformity of input forms.
  • Outputs: Examples include daily reminder lists and online volume statistics.
  • Processes: Examples include GPA calculations and system interfaces.
  • Performance: Examples include timely customer statements and class list generation.
  • Controls: Examples include security measures and approval processes.

4.4 Requirements Gathering (Fact Finding Activity)

Requirements gathering involves various techniques such as interviews, document reviews, observations, and surveys. It answers six key questions:

  • Who?
  • What?
  • Where?
  • When?
  • How?
  • Why?

The Process of Determining Requirements

Effective requirement determination requires collaboration between business and IT perspectives. Analysts must consider how to elicit requirements from stakeholders, with the understanding that this process evolves throughout the analysis phase.
