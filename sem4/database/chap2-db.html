Database Systems: Data Models (Module 2)

1. Data Modeling Fundamentals

  • Data Model: A simplified abstraction of real-world data structures (e.g., ER diagrams, relational tables).
  • Purpose: Acts as a blueprint for database design, ensuring consistency and communication between stakeholders.
  • Data Modeling Process: Iterative and progressive, involving entities, attributes, relationships, and constraints.

2. Core Components

Entities & Attributes

  • Entity: A distinct object (e.g., Customer, Invoice).
  • Attribute: A property of an entity (e.g., Customer_ID, Invoice_Date).
  • Naming Conventions: Prefix attributes with entity abbreviations (e.g., CUS_CREDIT_LIMIT).

Relationships & Constraints

Type Definition Example
1:1 One-to-One (e.g., Student ↔ Student ID) 1 employee manages 1 store.
1:M One-to-Many (e.g., Mentor ↔ Mentees) 1 department has many employees.
M:N Many-to-Many (e.g., Students ↔ Courses) Employees learn multiple skills.

Constraints: Rules like "GPA must be 0.00–4.00" or "Each class has one teacher."

3. Business Rules & Design

  • Definition: Clear policies that shape database structure (e.g., "An invoice belongs to one customer").
  • Role: Define entities, relationships, and constraints. Example:
    • "A customer may generate many invoices" → Customer and Invoice entities with a 1:M relationship.
  • Sources: Managers, policy documents, workflows.

4. Evolution of Data Models

Historical Models

  • Hierarchical Model (1960s): Tree-like parent-child structures (e.g., IBM’s IMS).
  • Network Model: Graph-based with schema/subschema, DML/DDL (e.g., CODASYL).

Modern Models

  • Relational Model:
    • Based on tables (relations), rows (tuples), and columns (attributes).
    • Example: AGENT and CUSTOMER tables linked via AGENT_CODE (see Fig 2.1).
  • Entity-Relationship Model (ERM):
    • Visualized using Chen, Crow’s Foot, or UML notations (Fig 2.3).
    • Example: A 1:M relationship between PAINTER and PAINTING.
  • Object-Oriented Model (OODM):
    • Combines data and methods (e.g., INVOICE object with attributes and behaviors).
    • Supports inheritance and class hierarchies.

Emerging Models

  • Big Data (3 Vs):
    • Volume (size), Velocity (speed), Variety (data types).
    • Tools: Hadoop, NoSQL (Cassandra, MongoDB).
  • NoSQL: Non-relational, distributed, and scalable for sparse/unstructured data.

5. Levels of Abstraction

Level Description Example
External End-user views (e.g., registration system for students). ER diagrams for specific departments.
Conceptual Global view, hardware/software independent. Unified ER model for the entire organization.
Internal DBMS-specific implementation (e.g., SQL table definitions). CREATE TABLE PROFESSOR (...)
Physical Storage details (e.g., disk blocks, indexes). File structures on a hard drive.

Independence: Logical (internal changes don’t affect conceptual) and Physical (storage changes don’t affect internal).

6. Summary & Exam Tips

  • Focus on translating business rules into ER diagrams.
  • Understand differences between hierarchical, relational, and NoSQL models.
  • Memorize the 3 Vs of Big Data and NoSQL characteristics.
  • Practice converting ER models to relational schemas.
